# B4: Incremental Authorization（機能別の遅延承認）の可否と設計

> 目的: Classroom 連携（`classroom.*`）やメール送信（`script.send_mail`）を、その機能を**初めて使うときにだけ**認可要求する構成にできるか検証し、実装または代替案を提示する。
>
> 受け入れ基準: 「最小コア機能は軽いスコープだけで開始できる」

## 結論（先に要点）

- **Apps Script のビルトイン承認では、真の Incremental Authorization は実現できません。** 単一スクリプトプロジェクトでは、マニフェスト `oauthScopes`（および全ソースコードの静的解析で検出される API 利用）に含まれる**すべてのスコープが初回認可時に一括同意**されます。コード内に `Classroom.*` / `MailApp.*` の参照がある限り、遅延実行のコードパスでもそのスコープは初回に要求されます。
- したがって「コア機能は軽いスコープだけで開始」を厳密に満たすには、**Classroom/メールを別 OAuth フロー（apps-script-oauth2 ライブラリ＋標準 GCP プロジェクト）で遅延承認する**設計が必要です。これは **C1（標準 GCP プロジェクトの紐付け）が前提**になります。
- 当面の現実解として、**使用時の認可チェック＋案内（graceful pre-flight）**を全「重い機能」に適用し、未使用ユーザーの体験を損なわないようにします（同意画面のスコープ自体は減りませんが、UX 上の破綻を防ぎます）。既にメール（`saveTaskReminderSettings`）で実装済みのパターンを Classroom にも展開できます。

## GAS の認可モデル（検証）

1. **スコープの決定**: GAS は「マニフェストの明示 `oauthScopes`」＋「ソースコードの静的解析で検出した API」からスコープ集合を確定します。
2. **同意タイミング**: 確定したスコープ集合は、ユーザーが初めてアプリを認可するとき（Web アプリなら初回アクセス時の同意画面）に**一括提示**されます。
3. **遅延できない理由**: `ScriptApp.getOAuthToken()` やトリガー実行はプロジェクト単位の同意に基づくため、「Classroom を使う関数を呼んだ瞬間にだけ classroom スコープを足す」ことはビルトインではできません。関数分割・遅延呼び出しをしても、静的解析でスコープが検出されるため効果がありません。
4. **`getAuthorizationInfo` の限界**: `ScriptApp.getAuthorizationInfo(AuthMode.FULL)` は「プロジェクト全体として再認可が必要か」を返すだけで、**スコープ単位の付与状況は判別できません**。よって「classroom だけ未承認」を正確に検出する API はありません。

## 現状のスコープ分類（本アプリ）

| 区分 | スコープ | 必要性 |
|------|----------|--------|
| コア（軽い） | `spreadsheets`, `drive.file`, `userinfo.email`, `script.scriptapp`, `script.external_request`, `script.container.ui` | 週案・単元・タスク・PDF取込（Picker）に必須 |
| 重い（機能別） | `classroom.courses.readonly`, `classroom.announcements` | Classroom 連携を使う人のみ |
| 重い（機能別） | `script.send_mail` | タスクのメールリマインダーを使う人のみ |

コア機能だけを使うユーザーにとって、Classroom/メールのスコープ同意は不要な離脱要因になり得ます。

## 実現方式の比較

### 方式A（推奨・本命）: 別 OAuth2 フローによる遅延承認
- **概要**: マニフェストの `oauthScopes` から `classroom.*` / `script.send_mail` を外し、これらの機能は
  [apps-script-oauth2 ライブラリ](https://github.com/googleworkspace/apps-script-oauth2) を使って、**その機能を初めて使うときに**別途 OAuth 同意（ポップアップ/リダイレクト）を発生させる。トークンはユーザー別に保存し、以後は再利用。
- **前提**: **標準 GCP プロジェクト（C1）**と OAuth クライアント（C2）が必要（デフォルトの GAS プロジェクトでは自前 OAuth クライアントを持てない）。
- **メリット**: コア初回同意を軽いスコープだけにできる（受け入れ基準を満たす）。機能単位で同意を分離できる。
- **デメリット**: 実装コスト大（トークン管理・再認可・失効処理）、Classroom は Advanced Service ではなく **REST 呼び出し**へ書き換えが必要（`Classroom.*` を使うと結局スコープが静的検出されるため）。同様にメールも `MailApp` ではなく Gmail REST もしくは別手段に。
- **審査**: 同意画面（C2）とスコープ用途説明（C4）は方式Aでも必要。

### 方式B: 2パッケージ配布（コア版／連携版）
- Classroom/メールを含む版と含まない版を別プロジェクトとして配布。コア版は軽いスコープのみ。
- **デメリット**: 配布・保守が二重化。共通URL運用（本ロードマップのゴール）と相性が悪い。→ 非推奨。

### 方式C（当面の現実解）: 使用時 pre-flight 認可チェック＋案内
- スコープ同意自体は一括のままだが、**重い機能を使う導線で認可状態を確認し、未承認なら分かりやすく案内**する。
- 既に `saveTaskReminderSettings`（`07_WebApp.gs`）で実装済み:
  ```js
  try {
    MailApp.getRemainingDailyQuota();       // 未承認なら例外（script.send_mail）
    Session.getEffectiveUser().getEmail();  // userinfo.email
  } catch (permErr) {
    throw new Error('メール送信の権限が不足しています。アプリを開き直して…承認してからお試しください。');
  }
  ```
- **Classroom 連携の入口**（コース一覧取得・投稿）にも同じ「try→分かりやすいエラー」パターンを展開すれば、
  未承認時の無言失敗を防げる。
- **限界**: これは「同意画面を軽くする」ものではない（受け入れ基準の厳密達成にはならない）。方式Aまでの UX 緩和策。

## 推奨ロードマップ

1. **いま（B4 の範囲）**: 方式C を全「重い機能」に適用し、未使用ユーザーの体験破綻を防ぐ（低リスク）。本書で方式Aの設計を確定。
2. **C1/C2 完了後**: 方式A を実装。マニフェストからは `classroom.*` / `script.send_mail` を外し、コア初回同意を軽量化。Classroom/メールは apps-script-oauth2 の遅延同意へ移行。
3. **C4（Google 検証）**: 方式A後のスコープ構成（コア=軽い、機能別=遅延）で申請すると、未使用ユーザーへの過剰要求がなくなり説明が通りやすい。

> **依存関係**: 本タスクの「実装」部分（方式A）は C1（標準 GCP プロジェクト）に依存するため、コード実装は C1 以降に回すのが妥当です。B4 の現時点の成果物は「方式C の適用方針」と「方式A の確定設計」です。
