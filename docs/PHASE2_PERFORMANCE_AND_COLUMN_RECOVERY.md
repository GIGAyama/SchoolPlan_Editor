# Phase 2: 体感速度・通信効率と列ずれ復旧

## 1. 列ずれの原因

旧 `getDbColumns()` は、データベースの列マップを Apps Script の `ScriptCache` に保存していました。
キャッシュキーにはシート名しか含まれていなかったため、マルチテナント環境で複数ユーザーが同じ
`データベース` というシート名を使うと、列順が異なる別スプレッドシートの列マップが共有される
可能性がありました。

その場合、スプレッドシート上のデータは移動していないのに、Webアプリが別の物理列を読み、
「宿題が行事に表示される」「単元と学習内容が入れ替わる」などの症状が発生します。

## 2. 修正内容

- ScriptCache を列位置の情報源として使用しない
- 接続中の実シートの1行目を読み、ヘッダー名から列を解決する
- 列の並び順を自由にする
- 全角数字、`1時間目`、`単元名1`、`内容1` など旧表記を正規化する
- 旧キャッシュの削除処理を残す
- 保存前に必要な列が解決できているか検証する

## 3. 既存データへの影響

この更新は、既存のセルを自動的に移動しません。

表示だけがずれていた場合、更新版をデプロイして再読み込みすると、実際のヘッダー位置に従って
正しく表示されます。

誤表示中にWebアプリから保存した場合は、誤った列へ値が書き込まれている可能性があります。
どの値が本来どこにあったかを一般的なルールだけで断定できないため、自動修復は行いません。
スプレッドシートの変更履歴またはバックアップから、該当期間だけ確認してください。

## 4. 列構成の診断

Apps Script エディタから `getDbSchemaDiagnosticsFromWeb()` を実行すると、次の情報を返します。

- 使用中のデータベースシート名・シートID
- 実際のヘッダー一覧
- 論理項目から物理列への対応
- 重複見出し
- 保存に必要だが見つからない列
- 保存可能かどうか

`safeToWrite: true` であれば、週案のV2保存APIが必要とする列を解決できています。

## 5. 性能改善

### 起動

従来は起動直後に、週案、単元マスタ、年間時数、設定、タスク、振り返り、複数学級設定などを
別々に通信していました。

Phase 2では以下に分割します。

1. 初期画面に必要な週案・単元候補・学級状態を1通信で取得
2. 初期描画後にタスク・振り返り・初期設定状態を1通信で取得
3. 年間時数・設定・PDFなどは対象タブを開いたときだけ取得

### 週案の読込

- 従来: データベース全行・全列を読み込み、対象週を検索
- Phase 2: 日付列だけを検索し、対象週の最大7行だけを読み込み

### 週案の保存

- 従来: データベース全体を `setValues()` で書き戻し
- Phase 2: 変更された日の週案列だけを書き戻し

これにより、年間データ量が増えた場合の待ち時間と、無関係な列を巻き込むリスクを抑えます。

## 6. デプロイ後の確認

1. 新しいバージョンとしてWebアプリをデプロイする
2. ブラウザまたはPWAを完全に再読み込みする
3. 過去に入力した週を3週程度開く
4. 行事、朝学習、1〜6校時、単元、学習内容、宿題、持ち物を確認する
5. 1セルだけ編集して保存し、再読み込み後も同じ欄に残ることを確認する
6. 学級切替を使う場合は、列順の異なる2学級でも確認する

## 7. ロールバック

問題があれば、GASのデプロイ管理画面から直前の安定バージョンへ戻せます。
このPhase 2更新自体はデータ移動を行わないため、表示確認だけで問題が見つかった場合は安全に
旧バージョンへ戻せます。
